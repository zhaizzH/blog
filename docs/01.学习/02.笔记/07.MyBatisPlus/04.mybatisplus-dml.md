# MyBatisPlus DML 增删改控制

## ID 生成策略

不同的表应用不同的 ID 生成策略：

- 日志：自增（1,2,3,4……）
- 购物订单：特殊规则（FQ23948AK3843）
- 外卖单：关联地区日期等信息（10 04 20200314 34 91）
- 关系表：可省略 ID

### @TableId 注解

| 名称 | @TableId |
|------|----------|
| 类型 | 属性注解 |
| 位置 | 模型类中用于表示主键的属性定义上方 |
| 作用 | 设置当前类中主键属性的生成策略 |
| 相关属性 | `value`（默认）：设置数据库表主键名称<br/>`type`：设置主键属性的生成策略，值参照 IdType 枚举值 |

### 策略详解

#### AUTO 策略

使用数据库 ID 自增：

```java
@TableId(type = IdType.AUTO)
private Long id;
```

**注意：** 必须确保数据库表设置了 ID 主键自增，否则无效。

#### INPUT 策略

用户手工输入 ID：

```java
@TableId(type = IdType.INPUT)
private Long id;
```

需要将表的自增策略删除，且必须手动设置 ID 值，否则会报错。

#### ASSIGN_ID 策略

雪花算法生成 ID（可兼容数值型与字符串型）：

```java
@TableId(type = IdType.ASSIGN_ID)
private Long id;
```

不需要手动设置 ID，如果手动设置则使用自己设置的值。生成的是 Long 类型的数据。

#### ASSIGN_UUID 策略

以 UUID 生成算法作为 ID 生成策略：

```java
@TableId(type = IdType.ASSIGN_UUID)
private String id;  // 主键类型不能是Long，应改为String
```

需要将表的主键类型设置为 varchar，长度大于 32。

#### 雪花算法

雪花算法（SnowFlake）是 Twitter 官方给出的算法实现，生成的结果是一个 64bit 大小整数：

![雪花算法结构](https://img.zhaizz.top/file/blog-website/mybatisplus/1631243987800.png)

- **1bit**：不用，最高位是符号位，固定为 0
- **41bit**：时间戳，毫秒级
- **10bit**：工作机器 ID，高位 5bit 是数据中心 ID（0-31），低位 5bit 是工作节点 ID（0-31），最多容纳 1024 个节点
- **12bit**：序列号，每个节点每毫秒从 0 开始累加，最多 4095，共可产生 4096 个 ID

### 策略对比

| 策略 | 优点 | 缺点 |
|------|------|------|
| NONE / INPUT | 灵活 | 需手动设置，容易出现主键冲突 |
| AUTO | 简单，数据库自增 | 不可作为分布式 ID，仅限单台数据库服务器 |
| ASSIGN_UUID | 分布式可用，保证唯一 | 32 位字符串，长度过长，不能排序，查询性能慢 |
| ASSIGN_ID | 分布式可用，Long 类型，可排序，性能高 | 与服务器时间有关，修改系统时间可能导致重复主键 |

### 简化配置

**全局主键策略配置：**

```yml
mybatis-plus:
  global-config:
    db-config:
      id-type: assign_id
```

**全局表前缀配置：**

```yml
mybatis-plus:
  global-config:
    db-config:
      table-prefix: tbl_
```

MP 会拿 `tbl_` 加上模型类的首字母小写组装成数据库表名。

## 逻辑删除

### 概念

- **物理删除**：业务数据从数据库中丢弃，执行 delete 操作
- **逻辑删除**：为数据设置是否可用状态字段，删除时设置状态字段为不可用状态，数据保留在数据库中，执行 update 操作

逻辑删除解决的问题：删除员工数据时，关联的合同数据不会成为无主数据，同时合同统计也不会丢失数据。

### @TableLogic 注解

| 名称 | @TableLogic |
|------|-------------|
| 类型 | 属性注解 |
| 位置 | 模型类中用于表示删除字段的属性定义上方 |
| 作用 | 标识该字段为进行逻辑删除的字段 |
| 相关属性 | `value`：逻辑未删除值<br/>`delval`：逻辑删除值 |

### 实现步骤

#### 步骤 1：数据库表添加 deleted 列

添加 `deleted` 列，默认值为 `0`（正常），`1` 代表删除。

#### 步骤 2：实体类添加属性

```java
@TableLogic(value="0", delval="1")
private Integer deleted;
```

#### 步骤 3：运行删除方法

逻辑删除最后走的是 update 操作：

```sql
UPDATE tbl_user SET deleted=1 WHERE id = ? AND deleted=0
```

### 逻辑删除对查询的影响

MP 的逻辑删除会将所有查询都自动添加一个 `deleted=0` 的条件，已被删除的数据不会被查询出来。

如果需要查询包含已删除的数据，可以使用原生 SQL：

```java
@Mapper
public interface UserDao extends BaseMapper<User> {
    @Select("select * from tbl_user")
    public List<User> selectAll();
}
```

### 全局配置

```yml
mybatis-plus:
  global-config:
    db-config:
      # 逻辑删除字段名
      logic-delete-field: deleted
      # 逻辑删除字面值：未删除为0
      logic-not-delete-value: 0
      # 逻辑删除字面值：删除为1
      logic-delete-value: 1
```

**逻辑删除的本质是修改操作。如果加了逻辑删除字段，查询数据时也会自动带上逻辑删除字段。**

## 乐观锁

### 概念

乐观锁主要解决的问题是：当要更新一条记录的时候，希望这条记录没有被别人更新。典型场景是秒杀业务。

### 实现思路

1. 数据库表中添加 `version` 列，默认值为 1
2. 第一个线程取出记录时，获取当前 version=1
3. 第二个线程取出记录时，获取当前 version=1
4. 第一个线程执行更新：`SET version = 2 WHERE version = 1`，成功
5. 第二个线程执行更新：`SET version = 2 WHERE version = 1`，此时 version 已经为 2，更新失败

不管谁先执行，都能确保只有一个线程更新数据成功。

### 实现步骤

#### 步骤 1：数据库表添加 version 列

列名为 `version`，默认值为 `1`。

#### 步骤 2：模型类添加对应属性

```java
@Version
private Integer version;
```

#### 步骤 3：添加乐观锁拦截器

```java
@Configuration
public class MpConfig {
    @Bean
    public MybatisPlusInterceptor mpInterceptor() {
        MybatisPlusInterceptor mpInterceptor = new MybatisPlusInterceptor();
        mpInterceptor.addInnerInterceptor(new OptimisticLockerInnerInterceptor());
        return mpInterceptor;
    }
}
```

#### 步骤 4：执行更新操作

正确做法是先查询再更新，确保携带 version 数据：

```java
@Test
void testUpdate(){
    //1.先通过要修改的数据id将当前数据查询出来
    User user = userDao.selectById(3L);
    //2.将要修改的属性逐一设置进去
    user.setName("Jock888");
    userDao.updateById(user);
}
```

模拟并发场景：

```java
@Test
void testUpdate(){
    User user = userDao.selectById(3L);     //version=3
    User user2 = userDao.selectById(3L);    //version=3
    user2.setName("Jock aaa");
    userDao.updateById(user2);              //version=>4
    user.setName("Jock bbb");
    userDao.updateById(user);               //version=3，条件不成立，更新失败
}
```

第二个更新操作因为 version 已经被第一个线程改为 4，所以 `WHERE version=3` 的条件不成立，更新失败。
